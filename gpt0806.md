# Prioritas P0 - Wajib Dibereskan Sebelum Ujian Massal

## 1. Race condition saat start ujian

### Lokasi kode

`app/Services/ExamService.php`

Flow saat ini:

```php
$existing = DB::table('sesi_ujians')
    ->where(['user_id' => $user->id, 'jadwal_ujian_id' => $j->id])
    ->first();

if ($existing) {
    return $existing;
}

$sessionId = DB::table('sesi_ujians')->insertGetId([...]);
```

### Masalah

Request paralel bisa membuat dua sesi untuk siswa dan jadwal yang sama.

Skenario:

1. siswa klik mulai dua kali
2. request A cek sesi belum ada
3. request B cek sesi belum ada
4. request A insert sesi
5. request B insert sesi
6. satu siswa punya dua sesi pada jadwal yang sama

### Dampak

- data sesi dobel
- hasil ujian tidak konsisten
- monitoring bisa kacau
- jawaban siswa bisa terhubung ke sesi yang berbeda

### Rekomendasi

Tambahkan unique index dan ubah flow menjadi transaction-safe.

### Solusi SQL

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_sesi_ujians_user_jadwal
ON sesi_ujians (user_id, jadwal_ujian_id);
```

### Solusi kode

Gunakan transaction dan tangani duplicate key:

```php
return DB::transaction(function () use ($user, $j, $ip, $ua) {
    $existing = DB::table('sesi_ujians')
        ->where('user_id', $user->id)
        ->where('jadwal_ujian_id', $j->id)
        ->lockForUpdate()
        ->first();

    if ($existing) {
        return $existing;
    }

    $sessionId = DB::table('sesi_ujians')->insertGetId([
        'user_id' => $user->id,
        'jadwal_ujian_id' => $j->id,
        'waktu_login' => now(),
        'status' => 'aktif',
        'ip_address' => $ip,
        'device_info' => json_encode(['user_agent' => $ua]),
        'last_seen_at' => now(),
        'created_at' => now(),
        'updated_at' => now(),
    ]);

    // insert sesi_ujian_soals di transaction yang sama

    return DB::table('sesi_ujians')->find($sessionId);
});
```

Catatan: unique index tetap wajib, karena `lockForUpdate()` pada row yang belum ada tidak cukup mencegah insert paralel.

## 2. Race condition simpan jawaban

### Lokasi kode

`app/Services/ExamService.php`

Flow saat ini:

```php
Redis::hset("queue_jawaban:{$s->id}", (string) $q->id, json_encode($payload));
PersistAnswerSnapshot::dispatch($s->id, $q->id)->onQueue('answers');
```

Lalu persist:

```php
DB::table('jawaban_siswas')->updateOrInsert(
    ['sesi_ujian_id' => $sessionId, 'bank_soal_id' => $soalId],
    [...]
);
```

### Yang sudah bagus

- jawaban masuk Redis lebih dulu
- field Redis per `bank_soal_id`, jadi latest answer menimpa answer lama
- Redis cleanup memakai Lua `forgetIfCurrent()` sehingga job lama tidak menghapus value baru

### Masalah

`updateOrInsert()` tidak atomic terhadap timestamp. Jika worker paralel, jawaban lama masih mungkin menimpa jawaban baru.

Skenario:

1. Job A membaca jawaban lama
2. Job B membaca jawaban baru
3. Job B menulis jawaban baru
4. Job A menulis jawaban lama
5. DB berisi jawaban lama

### Dampak

- jawaban akhir siswa bisa salah
- nilai akhir bisa salah
- audit menunjukkan urutan yang membingungkan

### Rekomendasi

Gunakan PostgreSQL atomic upsert dengan kondisi timestamp.

### Solusi SQL wajib

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_jawaban_siswas_session_soal
ON jawaban_siswas (sesi_ujian_id, bank_soal_id);
```

### Solusi upsert PostgreSQL

```sql
INSERT INTO jawaban_siswas (
    sesi_ujian_id,
    bank_soal_id,
    opsi_jawaban_id,
    jawaban_essay,
    tipe_soal,
    skor,
    scoring_status,
    client_updated_at,
    server_updated_at,
    created_at,
    updated_at
)
VALUES (...)
ON CONFLICT (sesi_ujian_id, bank_soal_id)
DO UPDATE SET
    opsi_jawaban_id = EXCLUDED.opsi_jawaban_id,
    jawaban_essay = EXCLUDED.jawaban_essay,
    tipe_soal = EXCLUDED.tipe_soal,
    skor = EXCLUDED.skor,
    scoring_status = EXCLUDED.scoring_status,
    client_updated_at = EXCLUDED.client_updated_at,
    server_updated_at = EXCLUDED.server_updated_at,
    updated_at = EXCLUDED.updated_at
WHERE jawaban_siswas.server_updated_at IS NULL
   OR EXCLUDED.server_updated_at >= jawaban_siswas.server_updated_at;
```

## 3. Submit ujian belum lock row sesi

### Lokasi kode

`app/Services/ExamService.php`

Flow:

```php
if ($s->status !== 'aktif') {
    return $s;
}

$this->flushAll($s->id);

$score = DB::table('jawaban_siswas')
    ->where('sesi_ujian_id', $s->id)
    ->sum('skor');

DB::table('sesi_ujians')->where('id', $s->id)->update([
    'status' => 'selesai',
    'nilai_akhir' => $score,
]);
```

### Masalah

Submit bisa terpanggil lebih dari sekali:

- siswa klik selesai
- browser retry
- auto-submit saat waktu habis
- API dan web request bersamaan

### Dampak

- event submit dobel
- nilai dihitung saat flush belum stabil
- status sesi update berulang

### Rekomendasi

Gunakan transaction dan lock row sesi.

### Solusi kode

```php
public function submit(object $s): object
{
    return DB::transaction(function () use ($s) {
        $session = DB::table('sesi_ujians')
            ->where('id', $s->id)
            ->lockForUpdate()
            ->first();

        if (!$session || $session->status !== 'aktif') {
            return $session ?: $s;
        }

        $this->flushAll($session->id);

        $score = DB::table('jawaban_siswas')
            ->where('sesi_ujian_id', $session->id)
            ->sum('skor');

        DB::table('sesi_ujians')
            ->where('id', $session->id)
            ->where('status', 'aktif')
            ->update([
                'status' => 'selesai',
                'waktu_submit' => now(),
                'nilai_akhir' => $score,
                'updated_at' => now(),
            ]);

        $this->event($session->id, 'submit');

        return DB::table('sesi_ujians')->find($session->id);
    });
}
```

## 4. Queue jawaban belum unique/debounce

### Lokasi kode

`app/Jobs/PersistAnswerSnapshot.php`

Job saat ini sangat sederhana:

```php
class PersistAnswerSnapshot implements ShouldQueue
{
    public function handle(ExamService $service): void
    {
        $service->flushOne($this->sessionId, $this->questionId);
    }
}
```

### Masalah

Setiap save jawaban membuat job baru. Jika siswa ganti jawaban pada soal yang sama berkali-kali, queue bisa penuh dengan job duplikat.

### Dampak

- queue backlog besar
- worker memproses job yang sudah tidak relevan
- latency persist meningkat

### Rekomendasi

Gunakan unique job per `(sessionId, questionId)`.

### Solusi kode

```php
use Illuminate\Contracts\Queue\ShouldBeUniqueUntilProcessing;
use Illuminate\Contracts\Queue\ShouldQueue;

class PersistAnswerSnapshot implements ShouldQueue, ShouldBeUniqueUntilProcessing
{
    use Dispatchable, InteractsWithQueue, Queueable, SerializesModels;

    public int $uniqueFor = 30;

    public function __construct(
        public int $sessionId,
        public int $questionId
    ) {}

    public function uniqueId(): string
    {
        return $this->sessionId . ':' . $this->questionId;
    }

    public function handle(ExamService $service): void
    {
        $service->flushOne($this->sessionId, $this->questionId);
    }
}
```

## 6. Ping online siswa update DB terus

### Lokasi kode

`WebController::ping()` dan `ApiController::ping()`

Flow:

```php
DB::table('sesi_ujians')
    ->where('id', $session->id)
    ->update(['last_seen_at' => now(), 'updated_at' => now()]);
```

### Masalah

Jika 1500 siswa ping setiap 10 detik, DB menerima sekitar 150 update/detik hanya untuk heartbeat.

### Dampak

- PostgreSQL write-heavy
- table bloat pada `sesi_ujians`
- monitoring online membebani ujian aktif

### Rekomendasi

Pindahkan heartbeat ke Redis.

### Solusi kode

```php
Redis::setex("cbt:session:online:{$session->id}", 45, now()->timestamp);
```

DB `last_seen_at` cukup diupdate setiap 60-120 detik atau saat event penting.

## 7. Reset session hard delete dan belum clear Redis

### Lokasi kode

`ManageController::resetSession()` dan `AdminApiController::resetSession()`

Flow:

```php
$this->clearDeviceAccessForUser((int) $session->user_id, Auth::id(), 'session_reset');
DB::table('sesi_ujians')->where('id', $id)->delete();
```

### Masalah

Session dihapus langsung, tapi pending answer Redis dan job lama bisa masih ada.

Key yang berisiko tertinggal:

```text
queue_jawaban:{sessionId}
```

### Dampak

- job lama bisa jalan setelah sesi dihapus
- orphan jawaban/audit
- histori ujian hilang
- debugging sulit

### Rekomendasi

Jangan hard delete sesi ujian. Gunakan status `reset` atau `invalidated`.

### Solusi kode

```php
DB::transaction(function () use ($id) {
    $session = DB::table('sesi_ujians')
        ->where('id', $id)
        ->lockForUpdate()
        ->first();

    abort_unless($session, 404);

    DB::table('sesi_ujians')->where('id', $id)->update([
        'status' => 'reset',
        'updated_at' => now(),
    ]);

    DB::table('jawaban_siswas')->where('sesi_ujian_id', $id)->delete();
    DB::table('sesi_ujian_soals')->where('sesi_ujian_id', $id)->delete();

    Redis::del("queue_jawaban:$id");
});
```

Tambahkan guard di `flushOne()`:

```php
$session = DB::table('sesi_ujians')->where('id', $sessionId)->first();
if (!$session || !in_array($session->status, ['aktif', 'selesai'], true)) {
    Redis::del("queue_jawaban:$sessionId");
    return;
}
```

## 10. ReportService aggregate jawaban global

### Lokasi kode

`app/Services/ReportService.php`

### Masalah

Subquery aggregate `jawaban_siswas` dibuat global, lalu baru di-join ke jadwal. Jika tabel jawaban besar, query report/PDF bisa berat.

### Dampak

- PDF lambat
- report membebani DB saat ujian
- arsip makin lambat seiring data bertambah

### Rekomendasi

Filter jawaban berdasarkan jadwal terlebih dahulu.

### Solusi kode

```php
$answers = DB::table('jawaban_siswas as a')
    ->join('sesi_ujians as s2', 's2.id', '=', 'a.sesi_ujian_id')
    ->where('s2.jadwal_ujian_id', $scheduleId)
    ->select(
        'a.sesi_ujian_id',
        DB::raw("coalesce(sum(case when a.tipe_soal='PG' then a.skor else 0 end),0) as nilai_pg"),
        DB::raw("coalesce(sum(case when a.tipe_soal='ISIAN' then a.skor else 0 end),0) as nilai_isian")
    )
    ->groupBy('a.sesi_ujian_id');
```

### Index pendukung

```sql
CREATE INDEX IF NOT EXISTS idx_sesi_ujians_jadwal_id
ON sesi_ujians (jadwal_ujian_id, id);

CREATE INDEX IF NOT EXISTS idx_jawaban_siswas_session_tipe
ON jawaban_siswas (sesi_ujian_id, tipe_soal);
```

## 12. Insert urutan soal pakai max(urutan)+1

### Lokasi kode

- `QuestionBankManagementController::import()`
- `AdminApiController::storeQuestion()`


### Solusi kode

```php
DB::transaction(function () use ($packageId, $data) {
    DB::table('paket_soals')
        ->where('id', $packageId)
        ->lockForUpdate()
        ->first();

    $next = (int) DB::table('bank_soals')
        ->where('paket_soal_id', $packageId)
        ->max('urutan') + 1;

    DB::table('bank_soals')->insert([
        'paket_soal_id' => $packageId,
        'urutan' => $next,
        // field lain
    ]);
});
```

```php
$latestSessions = DB::table('sesi_ujians')
    ->selectRaw('user_id, max(id) as latest_session_id')
    ->groupBy('user_id');

DB::table('users as u')
    ->leftJoinSub($latestSessions, 'ls', 'ls.user_id', '=', 'u.id')
    ->leftJoin('sesi_ujians as s', 's.id', '=', 'ls.latest_session_id');
```

```sql
CREATE INDEX IF NOT EXISTS idx_sesi_ujians_user_id_id
```

## 17. Archive pakai whereYear pada waktu UTC

## 18. Download PDF menandai sukses terlalu awal

### Lokasi kode

`ExamResultDownloadController::download()`

### Masalah

Download dicatat sebelum PDF benar-benar berhasil dibuat/dikirim.

```php
$pdf = $this->pdf($schedule, $class);

```

Generate PDF sebagai file tersimpan dengan status:

- `processing`
- `success`
- `failed`

Archive hanya boleh jika status `success`.

## 19. ImageService rawan memory spike

### Lokasi kode

`app/Services/ImageService.php`

### Masalah

Gambar dibaca penuh dan didecode ke bitmap tanpa validasi dimensi.

### Dampak

File kecil tapi resolusi besar bisa membuat PHP worker kehabisan memory.

### Solusi validasi

```php
'gambar' => ['nullable', 'image', 'max:2048', 'dimensions:max_width=2500,max_height=2500']
```

### Solusi guard service

```php
[$width, $height] = getimagesize($file->getRealPath());
abort_if($width * $height > 6_000_000, 422, 'Resolusi gambar terlalu besar.');
```

## 20. IdCodec bergantung APP_KEY

### Lokasi kode

`app/Services/IdCodec.php`

### Masalah

Hashids memakai `config('app.key')`. Jika APP_KEY berubah, hash URL lama tidak bisa didecode.

### Dampak

- link sesi gagal
- link jadwal gagal
- siswa bisa kena 404 saat ujian aktif setelah deploy salah

### Rekomendasi

Gunakan salt khusus.

### Solusi env

```env
ID_CODEC_SALT=stable_salt_jangan_diganti
```

### Solusi kode

```php
$this->hashids = new Hashids(config('app.id_codec_salt'), 8);
```
