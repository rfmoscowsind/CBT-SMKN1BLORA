# GPT08061 - Issue Penghambat Ujian, Resume Session, dan Toggle Device Lock

Dokumen ini adalah review lanjutan dari `gpt0806.md` dengan fokus:

1. issue yang bisa menghambat ujian berlangsung,
2. desain reset/unlock sesi agar siswa bisa lanjut tanpa kehilangan jawaban,
3. sisa waktu tetap dihitung server,
4. tambahan dashboard SuperAdmin untuk enable/disable device lock,
5. mode historis fingerprint saat device lock dimatikan.

Tanggal review: 2026-06-08.

Status implementasi: SELESAI pada 2026-06-08.

Dokumen implementasi teknis: `IMPLEMENTASI_GPT08061.md`.

Ringkasan implementasi:

- `Buka Sesi / Lanjutkan` sudah dipisah dari `Reset Ulang dari Nol`.
- Endpoint web non-destruktif: `POST /kelola/sesi/{id}/unlock-resume`.
- Endpoint API pengawas non-destruktif: `POST /api/v1/pengawas/sesi/{id}/unlock-resume`.
- `Reset Ulang dari Nol` tetap tersedia sebagai aksi destruktif eksplisit.
- `Device Lock ON/OFF` sudah tersedia di halaman manajemen fingerprint untuk SuperAdmin.
- Saat Device Lock OFF, fingerprint tetap dicatat ke histori tanpa memblokir siswa.
- Pending answer Redis tetap flushable saat sesi berstatus `terkunci`.

---

## Kesimpulan Singkat

Kode saat ini sudah cukup kuat untuk data safety utama:

- start ujian sudah memakai transaction dan lock,
- job jawaban sudah unique per sesi-soal,
- persist jawaban PostgreSQL sudah memakai atomic upsert berbasis timestamp,
- submit sudah memakai transaction dan lock,
- ping sudah Redis-first,
- production env example sudah mengarah ke Redis.

Namun untuk operasional ujian nyata, ada dua kebutuhan yang harus dipisah:

```text
1. Reset Ulang dari Nol
2. Buka Sesi / Lanjutkan
```

Reset ulang boleh menghapus jawaban. Tetapi untuk kasus siswa nyangkut, pindah device, browser error, atau fingerprint lock, jangan gunakan reset destruktif. Gunakan mode unlock/resume yang tidak menghapus jawaban dan tidak mengubah waktu mulai awal.

Tambahan baru:

```text
Device Lock juga perlu toggle global dari dashboard SuperAdmin.
Jika OFF, siswa boleh pindah device. Sistem tetap mencatat fingerprint historis berdasarkan NIS/NISN, tetapi tidak memblokir.
```

# A. Issue yang Bisa Menghambat Ujian Berlangsung

## 1. Reset sesi saat ini destruktif untuk kebutuhan lanjut ujian

Reset saat ini melakukan:

```php
Redis::del("queue_jawaban:$id");
DB::table('jawaban_siswas')->where('sesi_ujian_id', $id)->delete();
DB::table('sesi_ujian_soals')->where('sesi_ujian_id', $id)->delete();
DB::table('sesi_ujians')->where('id', $id)->update([
    'status' => 'reset',
    'waktu_submit' => null,
    'nilai_akhir' => null,
    'last_seen_at' => null,
    'updated_at' => now(),
]);
```

Dampaknya:

- jawaban DB hilang,
- pending answer Redis hilang,
- susunan soal sesi hilang,
- nilai sementara hilang,
- siswa mulai ulang dari nol.

Ini benar untuk **reset ulang ujian**, tetapi salah untuk kasus **siswa hanya nyangkut dan perlu lanjut**.

---

## 2. Start setelah status `reset` mengulang dari nol

Di `ExamService::start()`, jika sesi existing berstatus `reset` atau `invalidated`, sistem memakai ulang `sessionId`, tetapi menghapus jawaban, menghapus sesi soal, mengubah `waktu_login` menjadi `now()`, lalu membuat ulang `sesi_ujian_soals`.

Dampaknya:

```text
status reset = ujian ulang dari awal
```

Maka jangan pakai status `reset` untuk kasus lanjut ujian.

---

## 3. Status `terkunci` bisa menghambat siswa masuk soal

Jika device fingerprint membuat sesi `terkunci`, `start()` dapat mengembalikan sesi existing, tetapi halaman soal membutuhkan status `aktif`. Siswa bisa login tetapi tidak bisa mengerjakan jika status belum dibuka.

Solusi: buat endpoint `unlock/resume` yang mengubah status menjadi `aktif` tanpa menghapus jawaban dan tanpa mengubah `waktu_login`.

---

## 4. Pending answer Redis bisa hilang jika sesi terkunci sebelum flush

Saat ini `flushOne()` dan `flushAll()` melakukan drop Redis jika sesi dianggap tidak flushable:

```php
if (! $this->isFlushableSession($sessionId)) {
    Redis::del("queue_jawaban:$sessionId");
    return;
}
```

Jika status `terkunci` tidak dianggap flushable, maka pending answer bisa hilang.

Rekomendasi:

```php
private function isFlushableSession(int $sessionId): bool
{
    $status = DB::table('sesi_ujians')->where('id', $sessionId)->value('status');

    return in_array($status, ['aktif', 'selesai', 'terkunci'], true);
}
```

Atau lebih aman:

```php
private function shouldDropPendingAnswer(?string $status): bool
{
    return in_array($status, ['reset', 'invalidated'], true);
}
```

Jadi Redis pending answer hanya dibuang untuk status destructive, bukan karena status sementara.

---

## 5. Unique index wajib sudah berjalan di server

Karena kode sudah memakai `ON CONFLICT (sesi_ujian_id, bank_soal_id)`, PostgreSQL wajib punya unique index.

Checklist index:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_sesi_user_jadwal
ON sesi_ujians (user_id, jadwal_ujian_id);

CREATE UNIQUE INDEX IF NOT EXISTS uq_jawaban_session_soal
ON jawaban_siswas (sesi_ujian_id, bank_soal_id);

CREATE UNIQUE INDEX IF NOT EXISTS uq_sesi_soal
ON sesi_ujian_soals (sesi_ujian_id, bank_soal_id);

CREATE UNIQUE INDEX IF NOT EXISTS uq_sesi_nomor
ON sesi_ujian_soals (sesi_ujian_id, nomor_soal);
```

Cek duplikat sebelum migrate:

```sql
SELECT user_id, jadwal_ujian_id, COUNT(*)
FROM sesi_ujians
GROUP BY user_id, jadwal_ujian_id
HAVING COUNT(*) > 1;

SELECT sesi_ujian_id, bank_soal_id, COUNT(*)
FROM jawaban_siswas
GROUP BY sesi_ujian_id, bank_soal_id
HAVING COUNT(*) > 1;
```

# B. Desain Reset yang Benar

## 1. Reset Ulang dari Nol

Tujuan:

- siswa benar-benar mengulang ujian,
- jawaban lama dihapus,
- susunan soal bisa dibuat ulang,
- `waktu_login` baru,
- `nilai_akhir` null.

Nama tombol:

```text
Reset Ulang dari Nol
```

Wajib konfirmasi:

```text
PERINGATAN: Jawaban siswa akan dihapus dan siswa mengulang dari awal. Lanjutkan?
```

---

## 2. Buka Sesi / Lanjutkan

Tujuan:

- siswa yang nyangkut bisa login lagi,
- jawaban tetap aman,
- susunan soal tetap sama,
- `waktu_login` tidak berubah,
- sisa waktu tetap dihitung server,
- device fingerprint dibersihkan,
- status sesi kembali `aktif`.

Nama tombol:

```text
Buka Sesi / Lanjutkan
```

Perilaku wajib:

```text
jawaban_siswas      -> tetap, jangan dihapus
queue_jawaban Redis -> tetap, jangan dihapus
sesi_ujian_soals    -> tetap, jangan dihapus
sesi_ujians.id      -> tetap
waktu_login         -> tetap waktu mulai awal
nilai_akhir         -> jangan diubah sembarangan
status              -> aktif
fingerprint         -> clear
sisa waktu          -> dihitung server dari waktu_login awal + durasi, dibatasi waktu_selesai jadwal
```

Contoh method:

```php
public function unlockSessionForResume(Request $request, int $id)
{
    $this->authorizePermission('monitor-exams', ['SuperAdmin', 'Admin', 'Pengawas']);

    DB::transaction(function () use ($id) {
        $session = DB::table('sesi_ujians')
            ->where('id', $id)
            ->lockForUpdate()
            ->first();

        abort_unless($session, 404);
        abort_if($session->status === 'selesai', 422, 'Sesi sudah selesai. Tidak perlu unlock.');

        // Jangan hapus jawaban, Redis queue_jawaban, atau sesi_ujian_soals.
        $this->clearDeviceAccessForUser((int) $session->user_id, Auth::id(), 'session_resume_unlock');

        DB::table('sesi_ujians')->where('id', $id)->update([
            'status' => 'aktif',
            'waktu_login' => $session->waktu_login,
            'waktu_submit' => null,
            'last_seen_at' => null,
            'device_fingerprint' => null,
            'device_fingerprint_raw' => null,
            'is_device_locked' => false,
            'updated_at' => now(),
        ]);

        Redis::del("cbt:session:online:$id");
        Redis::del("cbt:ping:db:$id");
    });

    return response()->json([
        'success' => true,
        'message' => 'Sesi berhasil dibuka. Jawaban tetap aman dan siswa bisa melanjutkan ujian.',
    ]);
}
```

Route contoh:

```php
Route::post('/pengawas/sesi/{id}/unlock-resume', [ManageController::class, 'unlockSessionForResume'])
    ->name('pengawas.sesi.unlock-resume');
```

# C. Sisa Waktu Tetap Server-Side

Kode `remaining()` saat ini sudah cocok untuk kebutuhan server-side:

```php
$deadline = min(
    now()->parse($j->waktu_selesai),
    now()->parse($s->waktu_login)->addMinutes($j->durasi_menit)
);

return max(0, now()->diffInSeconds($deadline, false));
```

Syarat penting:

```text
waktu_login tidak boleh diubah saat unlock/resume.
```

Contoh:

```text
Jadwal: 08:00 - 10:00
Durasi: 60 menit
Siswa mulai: 08:10
Siswa nyangkut: 08:30
Pengawas unlock: 08:35
Siswa lanjut: 08:36

Deadline tetap 09:10
Sisa waktu saat 08:36 = 34 menit
```

# D. Tambahan Baru: Toggle Device Lock di Dashboard SuperAdmin

## 1. Kebutuhan

Di dashboard SuperAdmin bagian fingerprinting perlu ada satu tombol/switch global:

```text
Device Lock: ON / OFF
```

Tujuannya:

- saat ON, sistem bekerja seperti sekarang: device fingerprint dapat mengunci siswa jika pindah perangkat,
- saat OFF, siswa boleh pindah-pindah HP/laptop/browser tanpa diblokir,
- walaupun OFF, sistem tetap mencatat histori fingerprint/device untuk audit,
- pencatatan histori dikaitkan dengan identitas siswa, terutama NIS/NISN.

---

## 2. Perilaku saat Device Lock ON

```text
DEVICE_LOCK_ENABLED=true
```

Perilaku:

- fingerprint pertama siswa direkam,
- jika fingerprint berubah, sistem boleh lock user/sesi,
- pengawas dapat membuka dengan `Buka Sesi / Lanjutkan`,
- event perubahan device dicatat.

Cocok untuk ujian ketat/lab terkontrol.

---

## 3. Perilaku saat Device Lock OFF

```text
DEVICE_LOCK_ENABLED=false
```

Perilaku:

- siswa boleh pindah device,
- tidak ada blocking,
- tidak ada status `terkunci` hanya karena fingerprint berubah,
- sistem tetap mencatat histori device/fingerprint,
- histori disimpan berdasarkan key siswa: NIS/NISN/username/user_id,
- pengawas tetap bisa melihat jejak perangkat yang dipakai.

Mode ini cocok saat:

- ujian darurat,
- perangkat siswa bermasalah,
- banyak siswa memakai HP pribadi,
- browser sering reset fingerprint,
- ingin monitoring tanpa menghambat ujian.

## 5. Helper untuk membaca setting

Contoh helper/service:

```php
public function deviceLockEnabled(): bool
{
    return Cache::remember('setting:device_lock_enabled', 30, function () {
        $value = DB::table('app_settings')
            ->where('key', 'device_lock_enabled')
            ->value('value');

        if ($value === null) {
            return filter_var(env('DEVICE_LOCK_ENABLED', true), FILTER_VALIDATE_BOOLEAN);
        }

        return filter_var($value, FILTER_VALIDATE_BOOLEAN);
    });
}
```

Saat SuperAdmin mengubah toggle:

```php
DB::table('app_settings')->updateOrInsert(
    ['key' => 'device_lock_enabled'],
    [
        'value' => $request->boolean('enabled') ? 'true' : 'false',
        'updated_by' => Auth::id(),
        'updated_at' => now(),
    ]
);

Cache::forget('setting:device_lock_enabled');
```

## 6. Perubahan di logic fingerprint

Di `enforceDeviceFingerprintForUser()` atau logic sejenis, alurnya jangan langsung lock. Cek setting dulu.

Pseudo flow:

```php
$fingerprint = $this->makeDeviceFingerprint($request);
$this->recordDeviceFingerprintHistory($user, $request, $fingerprint, $session);

if (! $this->deviceLockEnabled()) {
    // Mode audit only: catat histori, tapi jangan lock.
    return;
}

// Mode lock aktif: baru enforce seperti sekarang.
if ($this->fingerprintMismatch($user, $fingerprint)) {
    $this->lockDeviceAccessForUser($user->id, $session?->id, 'device_mismatch');
    abort(423, 'Device berubah. Hubungi pengawas.');
}
```

Jadi saat OFF:

```text
record history = yes
lock = no
abort = no
```

## 7. Histori fingerprint berbasis NIS/NISN

Agar bisa dilacak, buat table histori khusus. Jangan hanya overwrite field fingerprint di `users`.

Contoh table:

```sql
CREATE TABLE IF NOT EXISTS device_fingerprint_histories (
    id bigserial PRIMARY KEY,
    user_id bigint NOT NULL,
    nis varchar(50) NULL,
    nisn varchar(50) NULL,
    username varchar(100) NULL,
    sesi_ujian_id bigint NULL,
    jadwal_ujian_id bigint NULL,
    fingerprint varchar(191) NOT NULL,
    fingerprint_raw jsonb NULL,
    ip_address inet NULL,
    user_agent text NULL,
    lock_enabled boolean NOT NULL DEFAULT true,
    action varchar(50) NOT NULL DEFAULT 'seen',
    created_at timestamp NOT NULL DEFAULT now()
);
```

Index yang disarankan:

```sql
CREATE INDEX IF NOT EXISTS idx_device_history_user_created
ON device_fingerprint_histories (user_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_device_history_nis_created
ON device_fingerprint_histories (nis, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_device_history_nisn_created
ON device_fingerprint_histories (nisn, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_device_history_session_created
ON device_fingerprint_histories (sesi_ujian_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_device_history_fingerprint_created
ON device_fingerprint_histories (fingerprint, created_at DESC);
```

Catatan:

- Jika field siswa di DB memakai `username` sebagai NIS/NISN, simpan juga `username`.
- Kalau ada kolom `nis` dan `nisn`, gunakan keduanya.
- Key tracking utama boleh `user_id`, tapi tampilan dashboard sebaiknya menampilkan NIS/NISN supaya mudah dicari pengawas.

## 8. Contoh fungsi pencatatan histori

```php
private function recordDeviceFingerprintHistory(User $user, Request $request, string $fingerprint, ?object $session = null): void
{
    DB::table('device_fingerprint_histories')->insert([
        'user_id' => $user->id,
        'nis' => $user->nis ?? null,
        'nisn' => $user->nisn ?? null,
        'username' => $user->username ?? null,
        'sesi_ujian_id' => $session?->id,
        'jadwal_ujian_id' => $session?->jadwal_ujian_id,
        'fingerprint' => $fingerprint,
        'fingerprint_raw' => json_encode($this->rawFingerprintPayload($request)),
        'ip_address' => $request->ip(),
        'user_agent' => $request->userAgent(),
        'lock_enabled' => $this->deviceLockEnabled(),
        'action' => $this->deviceLockEnabled() ? 'checked' : 'audit_only',
        'created_at' => now(),
    ]);
}
```

Untuk mengurangi write DB terlalu besar, boleh cache dedup 30-60 detik:

```php
$key = 'device-history:'.$user->id.':'.sha1($fingerprint.$request->ip().$request->userAgent());

if (Cache::add($key, 1, now()->addSeconds(60))) {
    $this->recordDeviceFingerprintHistory($user, $request, $fingerprint, $session);
}
```

## 10. Endpoint toggle

Contoh route:

```php
Route::post('/superadmin/device-lock/toggle', [DeviceLockSettingController::class, 'toggle'])
    ->middleware(['auth', 'can:manage-system-settings']);
```

Controller:

```php
public function toggle(Request $request)
{
    abort_unless(Auth::user()?->role === 'SuperAdmin', 403);

    $data = $request->validate([
        'enabled' => ['required', 'boolean'],
    ]);

    $old = app(DeviceLockSettingService::class)->deviceLockEnabled();

    DB::table('app_settings')->updateOrInsert(
        ['key' => 'device_lock_enabled'],
        [
            'value' => $data['enabled'] ? 'true' : 'false',
            'updated_by' => Auth::id(),
            'updated_at' => now(),
        ]
    );

    Cache::forget('setting:device_lock_enabled');

    DB::table('audit_logs')->insert([
        'user_id' => Auth::id(),
        'action' => 'device_lock_setting_changed',
        'payload' => json_encode([
            'old' => $old,
            'new' => (bool) $data['enabled'],
        ]),
        'created_at' => now(),
        'updated_at' => now(),
    ]);

    return response()->json([
        'success' => true,
        'device_lock_enabled' => (bool) $data['enabled'],
    ]);
}
```

# F. Catatan Akhir

Untuk ujian nyata, fitur fingerprint jangan hanya dianggap sebagai kunci keras. Lebih aman jika punya dua mode:

```text
Device Lock ON  = blokir jika device berubah
Device Lock OFF = catat histori saja, jangan blokir
```

Mode OFF sangat penting untuk kondisi darurat agar ujian tetap berjalan walaupun siswa pindah perangkat. Dengan histori berbasis NIS/NISN, pengawas tetap bisa melakukan audit setelah ujian tanpa menghambat pengerjaan siswa.
